## 技術的アプローチ

### ファイル構成（新規作成）
```
ai_interview-analysis/
├── src/
│   ├── app.py                    # メインエントリーポイント（既存を改修）
│   └── components/               # 新規ディレクトリ
│       ├── __init__.py
│       ├── styles.py            # カスタムCSS定義・注入
│       ├── ui.py                # UIヘルパーコンポーネント
│       └── charts.py            # チャート生成（Plotly）
└── tests/
    └── test_components/         # 新規テストディレクトリ
        ├── test_ui.py           # UIコンポーネントのテスト
        ├── test_charts.py       # チャートのテスト
        └── test_styles.py       # CSSのテスト
```

### テスト戦略
- **UIコンポーネント**: ヘルパー関数の単体テスト（入力→出力のテスト）
- **チャート**: データ変換ロジックのテスト
- **CSS**: CSS注入の正常性テスト
- **視覚的テスト**: マニュアルレビュー + スクリーンショット比較（オプション）

## リスクと制約

### 技術的リスク
- **Streamlitの制約**: カスタムCSSの制限、レイアウトの柔軟性
  - 対策: st.markdown + unsafe_allow_html でカスタムHTML/CSSを注入
- **既存機能の破壊**: リデザイン時に既存機能が動かなくなる可能性
  - 対策: 段階的リデザイン、各マイルストーンでの動作確認
- **レスポンシブ対応の難しさ**: Streamlitのグリッドシステムの制限
  - 対策: st.columns の比率調整、モバイルでは縦積み

### リソース制約
- **開発期間**: 3〜4日での完了を目指す
  - 対策: 優先順位を明確にし、必須機能から実装
- **デザインスキル**: Figmaなどのデザインツールなし
  - 対策: Linear、Notion、Vercelなどの既存SaaSを参考にする
